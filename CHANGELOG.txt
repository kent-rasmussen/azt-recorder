1.46.39

  text variant that can appear between the sync icon and the
  gear icon in the top bar: prefixes (timestamp variants vs

1.46.38
  Sync-status badge no longer gates the LANOK / +u/a suffix on
  uncommitted work. Pre-fix, _sync_status_info short-circuited
  to "not backed up" whenever last_sync == 0, conflating two
  separate things — committed history vs the working-tree dirty
  count — and hiding the LAN-share signal behind the same
  branch that handled "never pushed." Per CLIENT_INTEGRATION.md
  § 17b the suffix is a statement about committed history
  (commits_ahead vs unshared_commits); it's now computed
  whenever commits exist, regardless of last_sync. Uncommitted
  work (n_changes red) rides separately on the prefix when no
  commits are present, so "not backed up [red]+5[/red]" still
  flags dirty files but doesn't get a misleading "(OK)" suffix.
  Renders e.g. "not backed up (LANOK +3)" when github hasn't
  seen the project but a paired phone has all three local
  commits.

1.46.37
  Sync-status suffix now encodes the LAN-shared breakdown per
  azt_collab_client/projects.py:130-138 (ProjectStatus.
  unshared_commits, daemon 0.45.0+):
    commits_ahead == 0                              → "OK"
    commits_ahead > 0, unshared == 0                → "LANOK +N"
      (N ahead of github but all already on LAN — device could
       be wiped without losing data)
    commits_ahead > 0, 0 < unshared < commits_ahead → "+M/N"
    commits_ahead > 0, unshared == commits_ahead    → "+N"
  Pre-0.45 daemons omit unshared_commits; getattr default ==
  commits_ahead biases to the legacy "+N" rendering so older
  daemons keep working unchanged. Composes with the work_offline
  × lan_allow_sync suffix from 1.46.36, so a LAN-only project
  with one local-only commit renders "(LANOK +1) · LAN-only".

1.46.10
  Fix the cache-status duplicate-poller pattern flagged by the
  daemon team (status RPCs at ~2 Hz, "cache warm" logged twice,
  overlapping daemon round-trips). Root cause: _tick_cache_status
  fires every 1s and spawns a worker thread that calls
  cawl_cache_status async, then marshals back to the main thread
  to run _apply_cache_status — which is also where the event
  cancel happens. If the worker took >1s (typical during a
  cache-fill burst), tick N+1 spawned a second worker before the
  first's cancel ran, and both reached "cache warm" before the
  event was unscheduled. Added _cache_tick_in_flight to gate
  overlapping workers, and a _cache_status_warmed latch in
  _apply_cache_status so any late warm dispatch from a worker
  started before cancel ran becomes a no-op. Both flags reset
  in _start_cache_status_poll alongside the existing per-poll
  state so project switches start clean.

1.46.5
  _handle_pick first-setup detector now reads peer-side state
  (self.recorder is not None) instead of treating an empty
  last_project() as proof of first-setup. § 5's literal pattern
  was written for the brand-new-install case where both answers
  empty in lockstep, but daemon-side flows (e.g. Switch-Project)
  can transiently empty last_project() while the peer is mid-
  session — in which case the old detector treated a routine
  picker cancel as a first-setup-cancel and called self.stop(),
  exactly the § 14a anti-pattern. New shape: self.recorder is
  None → genuine first-setup → exit per § 5; otherwise return
  silently and leave the previous project loaded.

  Resolves the § 5 vs § 14a tension: stop() now fires only when
  there is literally nothing loaded yet, never as a reload
  mechanism a last_project blip could trip.

1.46.2
  CLIENT_INTEGRATION.md § 17c conformance (peer concurrency
  obligations). Closes the field-log signature the contract
  section was written to fix: parallel POST /sync calls per
  user tap, daemon refusing later ones with S.BUSY.

  Rule 1 — Single in-flight guard per (RPC, project). New
  shared dict App._sync_in_flight gates BOTH do_sync's
  sync_project AND _auto_commit_sync's commit_project on the
  same langcode key. Set on dispatch, cleared in a finally on
  the callback. The JOB_INTERRUPTED retry path holds the flag
  across the retry so a user tap during the retry window is
  dropped (not queued). Drop, do NOT queue — the in-progress
  sync covers what the dropped one would have done.

  Rule 3 — 250 ms peer-side debounce on do_sync. Tracks last
  tap time per langcode via time.monotonic(). sync_project is
  not server-side debounced (it's the explicit "do it now"
  gesture), so this is the peer's job.

  Rule 4 — Background sync-status poll tightened from 30 s to
  10 s. Contract range is 5-15 s; 30 s let daemon-driven
  pushes go invisible for up to half a minute after they
  drained. Build-time and on_resume re-arms both use 10 s.

  Net effect for the user: rapid Sync taps no longer pile up
  S.BUSY refusals; a successful daemon-driven push reaches
  the badge within 10 s instead of 30 s; the JOB_INTERRUPTED
  retry remains transparent.

1.46.1
  Recording-stop path: defer window.clearFlags(FLAG_KEEP_SCREEN_ON)
  onto the Kivy main thread. The add side (_start_android_recording)
  already deferred via Clock.schedule_once; the clear side
  (_stop_android_recording) was inlined, so when stop ran on

  touch its views"). Cosmetic until now — the try/except caught
  the throw and only the screen-on flag stayed set — but the
  fix matches the documented pattern in
  feedback_kivy_view_mutation_main_thread.

1.45.0
  Removed the auto-sync mode setting and migrated to the daemon's
  new commit/push split (paired with daemon 0.43.0 / client 0.43.0,
  CLIENT_INTEGRATION.md § 17b).

  Setting removed: was a 3-way selector on the config screen
  ('every swipe' / 'when idle' / 'manual only'). Behavior is now
  unconditional — every swipe fires commit_project, marking the
  boundary between meaningful chunks of work; the daemon's
  scheduler-drain loop decides when to push (online +
  post_online_grace_s + !work_offline). Commit and push are
  separate concerns — commit is local-state hygiene that should
  happen on every swipe regardless of connectivity, push is the
  network step and is the daemon's call. The user-facing toggle
  was conflating the two and exposing a knob users shouldn't have
  had to think about.

  Concretely removed: KV auto_sync_row + auto_sync_btn (config
  screen); _AUTO_SYNC_LABELS, _AUTO_SYNC_DEFAULT, _AUTO_SYNC_IDLE_SEC
  class attrs; _build_auto_sync_row, _show_auto_sync_overlay
  methods; the mode/idle branches in _auto_commit_sync. Stale
  peer_pref('auto_sync_on_swipe') values in $AZT_HOME/config.json
  are now ignored (no migration needed — unknown keys are silently
  dropped by the prefs store).

  Contract migration (per CLIENT_INTEGRATION.md § 17 / § 17b):

  - _auto_commit_sync now calls commit_project(langcode) instead
    of request_sync(langcode). The alias still works but
    commit_project is the explicit name for the commit-only RPC.
  - _on_sync_done (manual Sync button) gains an
    S.WORK_OFFLINE_ENABLED branch — toast + open_server_ui() so
    the user can flip the daemon-wide work-offline toggle. Same
    shape as the AUTH_REQUIRED / CONTRIBUTOR_UNSET routings.
  - _sync_status_info now reads ProjectStatus.work_offline and
    appends ' · offline' to the indicator when set. Examples:
    'HH:MM (+3) · offline' = three pending commits, daemon won't

  - Comments / docstrings referencing request_sync updated to

1.42.31
  Contract catch-up: CLIENT_INTEGRATION.md § 19 resolved the
  hook/contract drift filed in 1.42.30. The daemon team's
  resolution kept the hook injecting both the receiver and
  KILL_BACKGROUND_PROCESSES into every suite APK (including
  peers), but the rationale changed: each APK's own
  SuiteSelfReplaceReceiver reaps its OWN old-code processes
  via killBackgroundProcesses(getPackageName()) — no
  cross-package killing happens, so the permission is needed
  on the APK whose receiver fires.

  Implications for the recorder:
    - buildozer.spec.tmpl: KILL_BACKGROUND_PROCESSES stays
      OUT of android.permissions per § 19 "Peers MUST NOT add
      KILL_BACKGROUND_PROCESSES to their own android.
      permissions" — the hook adds it during manifest
      injection. Declaring it peer-side is redundant and
      gives Play Store an extra permission to scrutinise.
    - The hook-injected KILL_BACKGROUND_PROCESSES + receiver
      visible in our build output ([hook] injected … lines)
      is the expected behaviour now, not the contract drift
      it appeared to be when I filed the NOTES item.
    - bootstrap._kill_server_background (the pre-0.41.30
      peer-side cross-package backstop) is gone in client
      0.41.28+. Standard bootstrap() flow inherits the new
      shape automatically.

  No code change in the recorder this version — recording
  the catch-up + ratifying our existing spec state.

1.42.27
  Extended the NOTES_TO_DAEMON.md package-upgrade item with two
  clarifications under "Server APK self-handling":
    - Install channels covered: all four pathways (adb install
      -r, file-manager sideload, browser-driven INSTALL_PACKAGE,
      Google Play) trigger MY_PACKAGE_REPLACED — they converge
      on the same PackageInstaller commit. Fresh installs fire
      PACKAGE_ADDED instead, fine for this bug.
    - Manifest receiver, not runtime: runtime receivers require
      a live process; manifest receivers cold-start to deliver
      the broadcast, which is exactly what's needed when the OS
      has killed the old process as part of the replace.

1.42.26
  Filed NOTES_TO_DAEMON.md item: "Suite-wide package-upgrade
  handling — APK replacement vs. running process". Triggered by
  a user-visible bug where sideloading the required server APK
  in response to a client_too_old prompt didn't fix the prompt
  — Android kept the old server process alive serving the old
  version. Note lays out the principled fix:
    1. Server APK self-handles MY_PACKAGE_REPLACED → killProcess

    2. Peer "Check again" backstop with
       killBackgroundProcesses(<server package>).
    3. KILL_BACKGROUND_PROCESSES added to CLIENT_INTEGRATION.md
       § 2's required permissions list.
    4. Peer self-replacement (each peer handles its own
       MY_PACKAGE_REPLACED).
    5. Contract codification as CLIENT_INTEGRATION.md § 19.

  Peer-side bits will land here once the daemon team commits to
  the contract — held off on shipping the recorder's permission
  + receiver until then so we don't drift ahead of an evolving
  contract.

1.42.1
  Presplash diagnostic fixed (third attempt — this one works).
  1.41.32's switch to BitmapDrawable.getBitmap().getDensity() /
  .getWidth() was based on the assumption that those would report
  the source folder's density and native width. They don't:
  getDrawable() returns a bitmap that Android has *already
  pre-scaled* to the device target density, and the bitmap's
  density field is reset to target after scaling. So both fields
  reflect post-scaling state. Real-device logs that proved it:

    [presplash] device densityDpi=400 (xxhdpi); loaded unknown
        variant (source=400dpi, native_w=800)
    [presplash] device densityDpi=180 (hdpi);  loaded unknown
        variant (source=180dpi, native_w=360)

  Reported source_dpi == device_dpi on both, and native_w /
  source_dpi == 2 on both — the same ratio every bucket has in
  generate_presplash.py. All bucket rows collapse.

  Correct API: BitmapFactory.decodeResource(resources, res_id,
  opts) with opts.inJustDecodeBounds=True + opts.inScaled=False.
  After the call, opts.outWidth is the unscaled native pixel
  width of the picked folder's file, and opts.inDensity is the
  source folder's density. inJustDecodeBounds skips the bitmap
  allocation; inScaled=False is what actually disables the
  pre-scaling step.

  NOTES_TO_DAEMON.md updated with the correction and an ask to
  apply the same diagnostic to the server_apk splash so the two
  APKs can be told apart in a combined logcat.

1.42.0
  Auto-degrading audio-quality ladder. The 4×2 audio profile cross
  (high/medium/low/very_low × MPEG_4/AAC_ADTS) is now walked
  automatically when a recording fails, instead of leaving the user
  to discover the picker and guess which profile their device can
  handle. Three failure paths feed the same degrader:

    - Synchronous start exception (_publish_start_failure).
    - Stop-time exception on Android (MediaRecorder.stop() throws —
      typically the moov-atom-missing MTK HAL wedge).
    - Post-stop file-size validation: on Android filesystem paths
      (not SAF content URIs), if the encoded file is below 25 % of
      (held_seconds × bitrate / 8), the take is discarded and the
      ladder drops one rung. Catches the silent zero-byte failure
      mode where stop() returns clean but the encoder produced
      nothing.

  Ladder order swaps container before dropping bitrate (high →
  high_aac → medium → ...) — container swap dodges the MTK MPEG_4
  HAL bugs without giving up fidelity; bitrate drop is permanent
  quality loss and only happens when both containers fail at a
  given tier.

  The ceiling is persisted as peer_pref('audio_quality_ceiling') so
  it survives reboots; the audio-quality picker hides profiles
  above the pinned ceiling. Bad takes are deleted from disk and
  never advertised into the LIFT — the policy is "don't fool the
  user into thinking a recording was good when we know it wasn't."

1.41.32
  Presplash diagnostic logic fixed. 1.41.31's reverse-lookup
  was using Drawable.getIntrinsicWidth/Height, but those return
  device-SCALED dimensions, not native pixel sizes — Android
  scales the bitmap by (device_dpi / source_dpi) when reporting
  intrinsic size. That made multiple buckets collapse to the
  same number on any given device:

  - On 180dpi (hdpi range): mdpi (320*1.125=360), hdpi
    (480*0.75=360), and the .jpg fallback (480*1.125=540 —
    actually different, but close) all looked like "unknown"
    to the old reverse-lookup that compared against native
    bucket sizes.
  - On 400dpi (xxhdpi range): xhdpi (640*1.25=800), xxhdpi
    (960*0.833=800), xxxhdpi (1280*0.625=800) all collapse.
    Genuinely indistinguishable from dimensions alone.

  Fix: cast Drawable → BitmapDrawable, get the Bitmap, read
  Bitmap.getDensity() (the *source* density Android tags it
  with based on which folder it came from) plus
  Bitmap.getWidth() (native pixel width). Those two together
  uniquely identify the bucket:

  - density=120, native_w=240   → ldpi variant
  - density=160, native_w=320   → mdpi variant
  - density=160, native_w=480   → .jpg fallback (legacy)
  - density=240, native_w=480   → hdpi variant
  - density=320, native_w=640   → xhdpi variant
  - density=480, native_w=960   → xxhdpi variant
  - density=640, native_w=1280  → xxxhdpi variant

  Log shape now:

      [presplash] device densityDpi=420 (xxhdpi); loaded
          xxhdpi variant (source=480dpi, native=960x1599)

  Old "unknown — possibly .jpg fallback" message gone; the
  message either identifies the variant or reports the raw
  (source_dpi, native_w) pair for unrecognised cases.

1.41.30
  #9 follow-up: multi-density presplash variants now wired into
  the build. Research outcome:

  Buildozer's `android.add_resources = <src>:<dest>` spec key
  forwards a `--add-resource` flag to p4a, which in turn passes
  `--resource <abspath>:<dest>` to the dist's build.py. That
  invokes `shutil.copytree(src, join('src/main/res', dest),
  dirs_exist_ok=True)`. Setting `dest = '.'` merges the contents
  of the source directory directly into the dist's res/ root.

  So the recorder's `presplash_variants/drawable-<bucket>/
  presplash.png` (one per ldpi/mdpi/hdpi/xhdpi/xxhdpi/xxxhdpi)
  ends up at the canonical Android resource path
  `res/drawable-<bucket>/presplash.png`. The existing
  `presplash.filename` keeps writing `res/drawable/presplash.jpg`
  as the universal fallback for buckets none of the variants
  cover; Android's resource system resolves
  `R.drawable.presplash` across the .jpg / .png extension
  boundary by base name (aapt2 builds one resource entry with
  six bucket configs + the .jpg default).

  Changes:
  - buildozer.spec.tmpl: added
    `android.add_resources = %(source.dir)s/presplash_variants:.`
  - buildozer.spec.tmpl: added `presplash_variants` to
    `source.exclude_dirs` so the variants don't get bundled
    redundantly into the Python data tarball alongside their
    role as res/ resources.
  - .gitignore: added `presplash_variants/` (regenerated on
    every version bump by the build prehook's
    `generate_presplash.py` call; no need to commit binary
    artefacts).

  Per-peer mechanism — the recorder's `presplash_variants/`
  lives in its own source.dir, so the shared
  `../azt-collab/android/` tree stays untouched. Sister peers
  (azt-viewer, future apps) opt in by adding the same line +
  their own `presplash_variants/` directory.

  Tecno KN4 (~280dpi, hdpi/xhdpi border) now gets a 480x800 /
  640x1066 splash; Pixel 7 Pro (~xxxhdpi) gets a 1280x2133
  splash; the old "stretch the 480x800 to whatever the screen
  is" pixelation goes away.

  "Be eager when you have room to" — adaptive CAWL prefetch.

1.41.27
  H2 implementation: MediaRecorder setup moved off the Kivy
  main thread.

  start_recording now returns immediately; a worker thread runs
  the blocking JNI chain (ContentResolver.openFileDescriptor +
  setOutputFormat + setOutputFile + prepare + start), which on
  slow MediaTek chips can take 150-400ms. Previously this
  blocked the touch loop, letting touch_up queue behind it —
  the 1.41.20 min-hold-gate caught most but not all of those
  races, because a slow-prepare-then-long-hold measured the
  wall-clock as a long hold while the actual audio buffer only
  got the tail.

  New state machine:
  - _start_pending: True while the worker is mid-setup. Gates
    re-entrant start_recording; gates stop_recording's normal
    path (touch_up during pending sets the cancel flag and
    returns).
  - _start_cancel_requested: set when stop_recording fires
    during the pending window. Worker checks it after mr.start()
    returns; if set, tears down the just-started MediaRecorder
    without writing the LIFT reference (same end-state as the
    min-hold-gate tap path).

  _record_started_at semantics changed: now stamped when
  mr.start() actually returned (in _publish_start_success),
  not when start_recording was called. The min-hold-gate
  therefore measures real recorded-audio duration, not
  wall-clock since touch-down.

  iOS / desktop paths use the same async dispatcher — typically
  <50ms prepare equivalents, so no functional change visible,
  but the state machine is uniform across all three platforms.

  Two new methods on RecorderController:
  - _start_worker: blocking JNI on worker thread, marshals via
    Clock.schedule_once.
  - _publish_start_success / _publish_start_failure: main-
    thread callbacks that mutate publish-relevant state.

  Test cases covered:
  - Hold long enough for prepare + audio → normal record + save.
  - Slow prepare, short hold (touch_up before publish) →
    teardown, no LIFT write, "Hold the button to record" toast.
  - Slow prepare, hold past publish, short remaining audio →
    min-hold-gate measures post-publish time; tap-not-hold
    toast.
  - prepare raises → _publish_start_failure resets state.
  - Re-tap during pending → gated by _start_pending; no-op.

1.41.21
  Three defensive mitigations for the "crash while holding the
  record button" report (Tecno KN4 / Helio G81 / Android 16 /
  HiOS, 4GB physical RAM):

  (N1) Lowered the Android AAC config from 256kbps/48kHz mono
  to 128kbps/44.1kHz mono. The new combo is the universally-
  supported voice-quality config across Android hardware
  encoders; the old aggressive one has documented HAL-level
  failures on mid-tier MediaTek chips. Voice fidelity at 128k
  mono is still well above linguistic-fieldwork threshold;
  encoder working buffer halves, which matters on a 4GB-RAM
  device under zram swap.

  (N3) Added MediaRecorder.setMaxDuration(60_000). Bounds the
  worst-case encoder working set and gives a natural stopping
  point on a held button. The platform-fired duration event
  becomes a natural stop; can be lifted/changed later if
  needed.

  (N5) Added FLAG_KEEP_SCREEN_ON on the window during
  recording (cleared in _stop_android_recording). A long
  held-button gesture looks like idle to aggressive OEM power-
  management (HiOS in particular has been observed to freeze
  Activities under sustained no-touch-event windows); pinning
  the screen on prevents the MediaRecorder surface from
  being yanked mid-capture.

  _warm_jnius_classes also warms
  android.view.WindowManager$LayoutParams on the main thread
  so the worker-side autoclass during recording doesn't race
  the PyJNI ClassLoader hazard.

  N2 (microphone foreground service for HiOS / Android 16 mic-
  policy revocation) is the bigger manifest change; held back
  for a separate release pending confirmation that N1+N3+N5
  doesn't already resolve the field report.

  Reminder for the next field report: the recorder writes its
  own crash log to ANDROID_PRIVATE/azt_recorder.log (or
  /sdcard/azt_recorder.log on the fallback path). [CRASH]
  tracebacks from sys.excepthook land in that file and the
  field user can pull it without adb.

1.41.20
  Two more record-button-tap-crash mitigations.

  (1) Minimum hold-time gate. RecorderController._MIN_HOLD_SEC
  (400ms) is checked at the top of stop_recording; under that,
  the in-flight MediaRecorder is torn down without writing the
  LIFT reference (the M4A is missing its moov atom anyway —
  unplayable). The user gets a coaching toast: "Hold the button
  to record" (+ French translation). 400ms is comfortably
  longer than MediaRecorder.prepare() on Helio-G80-class chips
  (~150-400ms) so a slow start path isn't misread as a short
  hold.

  (2) Deferred _notify_ui. start_recording / stop_recording
  used to call _notify_ui synchronously, which triggers
  refresh_recorder_ui → btn_area.clear_widgets() + re-add. On
  slow chips under memory pressure (4GB physical + zram swap)
  the widget swap has been observed to race the in-flight Kivy
  touch dispatch and dereference a freed RecordButton instance.
  Switched to Clock.schedule_once(_notify_ui, 0) so the touch
  handler returns cleanly before the widget rebuild.

  New translatable string:
      "Hold the button to record"
          → "Maintenez le bouton pour enregistrer"

1.41.19
  _stop_android_recording: split stop() and release() so
  release() runs in a finally clause regardless of whether
  stop() raised. Previously stop() throwing IllegalStateException
  (typical when the user taps rather than holds — the M4A has
  no moov atom by the time stop fires) skipped release(),
  leaving the MediaRecorder + mic + native buffer wedged until
  GC. On slow MTK chips (e.g. Helio G81 / Tecno KN4 field
  report) the wedged recorder has been observed to take down
  the subsequent record attempt.

  release() inside finally is itself try/except'd so a
  cascading failure doesn't propagate up and mask the
  underlying stop() error in logcat.

1.41.14
  Daemon-driven CAWL prefetch per CLIENT_INTEGRATION.md § 10
  update.

  Peer no longer iterates CAWLHandle on every basename to warm
  the cache. Instead:

  - Computes one variant per CAWL id (lift.py's resolver tracks
    `_paths` parallel to `_basenames`; new
    `LIFTDatabase.all_cawl_paths()` returns the sorted list).
  - Hands the list to ``cawl_prefetch(langcode, paths)`` once;
    the daemon iterates in its own background thread.
  - Polls ``cawl_cache_status`` at 1 Hz to drive the progress
    label.

  The point: when peer iterated, the daemon couldn't know the
  size of the working set so `cawl_cache_status` denominator
  was the whole index (every variant) — progress bar plateaued
  far short of "100%" with no way for the user to tell whether
  caching was done. Now the daemon knows exactly what the peer
  asked for; ``total`` reflects the actual work, ``cached``
  ticks up toward it cleanly.

  Removed: ``_prefetch_images_worker`` and its peer-side
  per-basename CAWLHandle loop (CAWLHandle stays in use for
  on-demand fetches of the displayed image — the resolver's
  ``_pull`` path).

  Logging now state-change-only per the contract: the 1 Hz
  poll only prints when (cached, total) actually moves, so
  logcat doesn't fill with identical lines during a long
  warm. ``_apply_cache_status``'s "total=0" branch also logs
  just once per poll session.

1.41.1
  Switch lift.py:_save to LiftHandle.atomic_open_write (new in the
  client contract). Fully closes the peer-side truncation bug we
  filed (the "1-entry post-merge LIFT" symptom) on BOTH transports:

  - Filesystem paths: every save is now tempfile + fsync +
    os.replace via the client's helper. Random-suffixed tempfile
    (no collision when two saves overlap in process), atomic
    rename. Replaces the old manual tmp+fsync+rename which used a
    fixed '.tmp' suffix that *could* collide.

  a plain open_write on URI (no protection against same-process

  was two FDs writing concurrently from offset 0, with one
  entry's serialisation mid-embedded in another). The client's

  both windows.

  Select Project / start_over now runs _auto_commit_sync in the
  same worker thread as pick_project, instead of synchronously on
  the Kivy main thread before spawning the worker. With a project
  loaded, _auto_commit_sync issues a request_sync RPC that on the
  Android Binder transport (and a busy daemon) can stall long
  enough that the tap appears to do nothing — the main thread
  blocks on the RPC, which also blocks the JNI startActivity
  dispatch the picker schedules on the same thread. Moving both
  into the worker frees the main thread so the picker Activity
  can come up immediately.

  template-download path off the recorder and into the daemon. The
  recorder no longer hard-codes the SILCAWL URL or owns the download
  thread; it asks the daemon for a project from a template and opens
  the resulting LIFT.
    - azt_collabd/projects.py: new ``create_from_template(template_url,
      vernlang, dest_dir)`` — downloads the URL (size-capped at 10 MiB,
      60 s timeout), atomically writes ``<vernlang>.lift`` into
      dest_dir via ``os.replace`` over a temp file, and registers the
      Project. Returns the Project.
    - azt_collabd/config.py: new ``default_template_url()`` (overridable
      via ``configure(template_url=...)`` or ``AZT_DEFAULT_TEMPLATE_URL``
      env var). Default still points at the SILCAWL.lift on
      kent-rasmussen/lift_templates/main.
    - azt_collabd/server.py: new ``POST /v1/projects/from_template``
      endpoint (synchronous, body {template_url?, vernlang, dest_dir},
      response ``{ok, project}`` or ``{ok: False, error}``); daemon
      bumped 0.8.0 → 0.9.0.
    - azt_collab_client/__init__.py: new
      ``create_project_from_template(vernlang, dest_dir, template_url='')``
      that returns a Project on success or None on failure. Client
      bumped 0.9.0 → 0.10.0; ``MIN_SERVER_VERSION`` bumped 0.7.0 → 0.9.0
      so older daemons surface as ``server_too_old`` rather than 404
      on the new endpoint.
    - main.py: removed ``_SILCAWL_URL`` constant; ``new_from_template``
      now spawns a worker thread that calls
      ``create_project_from_template`` and on success schedules
      ``load_lift(project.lift_path)`` on the Kivy main thread. On
      failure dismisses the loading overlay and surfaces an error.

1.22.12
  Collaboration refactor — step 12 (LIFT-aware merge driver)
    - New azt_collabd/lift_merge.py: three_way_merge(base, ours,
      theirs) → MergeResult(merged_bytes, conflicts). Per-entry
      granularity in v1; per-field is a future improvement.
        * Entry only on one side, no base → keep.
        * One side modified, other unchanged → take modified.
        * Both modified to same canonical XML → take it.
        * Both modified differently OR add/add diverge OR
          modify/delete → keep both copies, each tagged with
          <annotation name="azt-lift-conflict" value="ours|theirs">.
          The "theirs" version gets a "-theirs" guid suffix to keep
          the document valid.
    - sync_repo rewritten:
        * Stage + commit local changes BEFORE fetching so they
          form a real commit on local <branch>.
        * porcelain.fetch instead of porcelain.pull.
        * Fast-forward when local is an ancestor of origin/<branch>.
        * Skip when origin is an ancestor of local.
        * Otherwise call _merge_diverged for a three-way merge:
            - .lift files → lift_merge.three_way_merge
            - other files → take changed side, prefer ours on tie,
              record a 'non-lift-modify-modify' conflict.
            - merge commit authored as the bot, with
              porcelain.commit(merge_heads=[remote_sha]). Falls back
              to grafting a second parent for older dulwich versions.
        * Push, with a retry loop (sync.merge_retry_max from settings)
          for races.
    - New helpers in repo.py: _walk_tree, _find_merge_base
      (uses dulwich.graph.find_merge_base when available; BFS
      fallback otherwise), _commits_between (limit 20), _is_ancestor.
    - New status code CONFLICTS (mirrored client-side); translation
      added: "Merge conflicts in {paths}".

1.36.3 — Align db.vernlang with server langcode on every load
================================================================

Why: User reported RecorderController.progress_text was showing
the wrong langcode. Tracing it: progress_text uses
self.db.vernlang. db.vernlang is set in two places — at LIFT
parse time (from observed lexical-unit lang attrs, which is
ambiguous when the file has multiple) and via db.set_vernlang().
load_lift's "apply langcode" block was checking _pending_vernlang
(set by _handle_pick) → fell back to peer_pref('vernlang'). The
auto-load path (_auto_load_last_project) sets _current_langcode
but NOT _pending_vernlang, so on every cold start the langcode
came from peer_pref — which is per-peer cached state and drifts
from the suite-wide last_project (the daemon's authoritative
key) when peers are switched or first-run.

The fix elevates _current_langcode to the primary signal in
load_lift and _reload_and_restore. clean_template() is now tied
ONLY to _pending_vernlang (its real meaning — "new-from-template
file, also tidy stub forms"), so the auto-load path never
triggers an unintentional clean_template _save() rewrite.

--- main.py: load_lift ---
- pending = getattr(self, '_pending_vernlang', '')
- if pending:
-     db.set_vernlang(pending)
-     db.clean_template()
-     self._pending_vernlang = ''
-     set_peer_pref('vernlang', pending)
- else:
-     saved_vern = peer_pref('vernlang', '') or ''
-     if saved_vern:
-         db.set_vernlang(saved_vern)
+ authoritative = getattr(self, '_current_langcode', '')
+ pending = getattr(self, '_pending_vernlang', '')
+ if authoritative:
+     db.set_vernlang(authoritative)
+     set_peer_pref('vernlang', authoritative)
+ elif pending:
+     db.set_vernlang(pending)
+     set_peer_pref('vernlang', pending)
+ else:
+     saved_vern = peer_pref('vernlang', '') or ''
+     if saved_vern:
+         db.set_vernlang(saved_vern)
+ if pending:
+     db.clean_template()
+ self._pending_vernlang = ''

--- main.py: _reload_and_restore ---
- saved_vern = peer_pref('vernlang', '') or ''
- if saved_vern:
-     db.set_vernlang(saved_vern)
+ authoritative = getattr(self, '_current_langcode', '')
+ if authoritative:
