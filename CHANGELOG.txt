CHANGELOG — A-Z+T Recorder

1.36.3
  Align db.vernlang with the server-supplied langcode on every
  load path, not just picker selections. Auto-loaded projects
  (_auto_load_last_project) had _current_langcode set but were
  falling through to peer_pref('vernlang') — which is per-peer
  cached state that can drift from the suite-wide last_project.
  RecorderController.progress_text consequently rendered the
  cached pref instead of the project's actual langcode. load_lift
  and _reload_and_restore now prefer _current_langcode and only
  fall back to the peer pref when no server-authoritative answer
  is in scope. clean_template() is decoupled from the vernlang-
  apply path; it only runs when _pending_vernlang is set (its
  real semantic — new-from-template).

1.36.2
  Removed the "Pick a project to continue." modal that blocked the
  recorder while the picker spawned. start_over now calls
  pick_project() on a worker thread directly; the result still
  lands in _handle_pick on the Kivy main thread. Helper methods
  _show_pick_overlay / _cancel_pick_overlay / _hide_pick_overlay
  and the _pick_cancelled flag are gone — cancellation is signalled
  by the picker itself returning err='cancelled' (already handled
  in _handle_pick).

1.36.1
  Walking back the cold-start changes from 1.36.0. When no project
  is loaded on launch, _auto_load_last_project goes back to firing
  the picker (start_over) — that's the right behaviour, the user
  doesn't want Settings on first run. The X button on Settings and
  on Collab also restore their pre-1.36.0 fallback to start_over
  when no project is loaded. The Select Project button stays — but
  is moved to the bottom of Settings (below Setup Collaboration),
  styled with BTN_INACTIVE to signal it's a rare action, not a
  primary one. Replaces the old (project-only) Start over button
  that lived in db_nav_box.

1.36.0
  Cold-start flow no longer fires the project picker uninvited.
  When _auto_load_last_project can't resolve a project (no saved
  langcode, daemon doesn't know it, or open_project fails), the
  app now lands on Settings instead of calling start_over() and
  blocking the user with the "Pick a project to continue." modal
  + the server-APK picker Activity. Settings grew a top-level
  "Select Project" button (always visible) so the user can change
  projects from there at any time. The redundant "Start over"
  button inside db_nav_box and the db_nav_box BoxLayout itself
  are removed. Settings X (apply_and_go) is a no-op when no
  project is loaded — it no longer punts back into the picker.
  CollabScreen close_collab also goes to Settings unconditionally.

1.35.1
  Settings UX: gloss-language picker is now a modal overlay instead
  of an inline collapsible grid. With many gloss languages the inline
  grid grew past three rows and shifted the visible region of the
  ScrollView so neighbouring rows fell off the top of the viewport.
  The overlay (matching _show_rec_overlay) sizes itself to the lang
  count, scrolls internally on tall lists, and never reflows the
  Settings layout. Active langs now appear as a dim summary next to
  the button (mirroring the Filter / Image-repository rows).

1.35.0
  P1 audit cluster: Android lifecycle, LIFT round-trip fidelity,
  and PyJNI thread-warming.
    - Android backgrounding mid-record no longer corrupts the M4A.
      LIFTRecorderApp grew on_pause / on_resume / on_stop hooks; the
      pause path runs stop_recording() so MediaRecorder writes its
      moov atom before the OS pauses us. on_pause returns True so
      Kivy doesn't fall through to on_stop, and on_resume is defined
      as required by the contract.
    - PyJNI ClassLoader-scope hazard mitigated. on_start eagerly
      autoclasses MediaRecorder / MediaPlayer / Uri / Intent /
      ParcelFileDescriptor / ByteArrayOutputStream / FileOutputStream
      / PythonActivity on the Kivy main thread, so subsequent
      worker-thread provider writes (image-download, picker callback,
      sync) hit cached proxies. (See feedback_pyjnius_threading.md.)
    - LIFT save no longer re-indents on every write. lift.py grew an
      _indent_dirty flag, set on new-element insertion (set_audio /
      set_illustration / _clean_forms) and cleared after save.
      Text-only edits (the common case: writing a new audio filename
      into an existing <text>) are now bit-stable in git, dramatically
      reducing diff churn and merge-conflict surface in azt_collabd.
    - LIFT namespace round-trip preserves original prefixes. lift.py
      now scans the source for xmlns:* declarations via iterparse
      before ET.parse and re-registers each prefix. FLEx-produced
      LIFT files no longer round-trip with ns0:flex= mangled prefixes.

1.33.1
  Data-integrity fixes around recording + LIFT save (audit findings
  B1, B3).
    - Recorder state no longer leaks across a failed start. start_recording
      computes the path, calls the platform start, and only flips
      _recording / _audio_path / _record_ok = True on success. A
      MediaRecorder.prepare() throw on URI projects used to write a
      bogus basename into <citation><form> because the URI short-circuit
      assumed the file existed; now the basename is written only when
      both start and stop ran clean.
    - _start_android_recording / _start_ios_recording / _start_desktop_recording
      re-raise on failure (after releasing the partial MediaRecorder
      and closing the pfd / sounddevice stream) instead of swallowing
      with print(). _start_native_recording returns the actual path
      used so extension-swapped paths (.m4a / .flac) flow back to the
      caller.
    - _stop_*_recording flip _record_ok = False on stop failure
      (Android IllegalStateException → no moov atom; desktop empty
      frames; iOS stop throw). The auto-play-on-stop schedule is
      skipped when the recording wasn't successful.
    - lift.py _save now writes filesystem-mode .lift files via a
      tmp-file + fsync + os.replace dance so a crash mid-write cannot
      truncate the lexicon. URI mode unchanged — the daemon's provider
      serialises concurrent writers on its side.

    - list_projects: dropped the legacy "also surface last_lift"

    - One-shot migration in build(): drop legacy prefs['last_lift']

    - _register_current_project / _auto_load_last_project: prefer

    - Requires azt_collab_client 0.19.0+ (recent.last_project /

1.32.0
  Tier 3: audio + image cross-package access on Android URI projects.
  Recording and playback now work end-to-end on projects opened from
  the standalone server APK; image additions remain daemon-owned but
  the recorder fetches sibling illustration files from the provider
  for display.
    - lift.py: new audio_target(basename) / image_target(basename)
      helpers wrap audio_uri_for / image_uri_for so callers stay
      blind to the path-vs-URI distinction. New _resolve_uri_image
      pulls sibling images via MediaHandle.open_read() into
      <image_cache_dir>/_uri_images/<href> on first render so Kivy
      AsyncImage can render by path; memoised per LIFTDatabase
      instance. _parse_entry routes through it on URI projects.
    - main.py _make_audio_path: no longer pre-creates audio_dir on
      URI projects (the daemon's provider auto-creates the dir on
      openFile('w')). Composes via db.audio_target so the same
      filename math produces a URI on Android URI projects and a
      path everywhere else.
    - main.py _start_android_recording: when db.is_uri, opens the
      audio URI via ContentResolver.openFileDescriptor('w') and
      hands the Java FileDescriptor to MediaRecorder.setOutputFile.
      Holds the ParcelFileDescriptor on self._record_pfd until
      _stop_android_recording closes it after release. Critical:
      we do NOT detachFd — the pfd owns the fd lifetime.
    - main.py play_audio: on URI projects, uses
      MediaPlayer.setDataSource(ctx, Uri.parse(audio_uri)) instead
      of the path-string overload; skips os.path.exists probing
      since the URI is resolved by the daemon's provider.
    - main.py stop_recording: skips the os.path.exists check on
      URI projects (daemon-served files don't appear at any
      filesystem path the peer can stat) and trusts the recorder
      stack to have written successfully.
    - main.py start_recording: removed the Tier-1+2 URI gate; URI
      audio writes now work.
    - main.py _save_remote_image / _download_and_set / _copy_and_set:
      gate messages updated to reflect that this is the architectural
      rule (daemon owns image additions on URI projects), not a
      Tier-3 TODO.
    - main.py _handle_pick: cache result.langcode as
      _current_langcode in addition to _pending_vernlang so
      _register_current_project skips the derive_langcode round-trip.
    - main.py _register_current_project: prefer the picker-supplied
      _current_langcode over derive_langcode. The langcode is
      server-supplied — the picker registers every emitted project
      (existing-tap / clone / new-from-template / open-file) with
      the daemon and threads the canonical projects.json key
      through the result. derive_langcode (itself a server RPC) is
      only the fallback for the recorder's startup auto-load from
      prefs['last_lift'], which skips the picker.
    - Requires azt_collab_client 0.18.0+ (MediaHandle, audio_uri_for,
      image_uri_for).

1.28.2
  Picker no longer hangs on Android with the modal overlay alone.
    - azt_collab_client 0.13.2 -> 0.13.3. _pick_project_android now
      dispatches all JNI work (autoclass, Intent build,
      resolveActivity, android.activity.bind, startActivityForResult)
      onto the Kivy main thread via Clock.schedule_once. The worker
      thread spawned by start_over() only waits for the result. Root
      cause: PyJNI's thread hook attaches worker threads to the JVM
      but does not install the app ClassLoader on them, so creating
      the ActivityResultListener proxy on the worker raised
      JavaException: ClassNotFoundException for
      org/kivy/android/PythonActivity$ActivityResultListener. The
      worker died before startActivityForResult ever ran, leaving the
      recorder showing only the "Pick a project to continue. Cancel"
      modal with no picker Activity behind it. No API change.
    - Recorder version bump only; no recorder code change beyond the
      version string. The fix is entirely in the symlinked client.
